GreenThumb Software Requirement Specifications (Version 1.0)

1. Introduction

1.1 SRS Purpose and Overview

This document covers the software requirements for the GreenThumb application, as well as its features. Its intended audience is the client, David Brown, as well as the GreenThumb team, who will refer to the document to create objectives and monitor progress. It is organized according to the IEEE Std 830.

1.2 SRS Scope

The product discussed in this document is GreenThumb, a mobile application that identifies plants using image recognition and allows guests and users to search a plant database. The application allows users to upload photographs of plants to expand the gallery of plants available for other users to see and expand the machine learning’s training material. The objective of the mobile application is to educate its users on with general species information.

2. Overall Description

2.1 Product Perspective

Similar apps to GreenThumb, such as the Florist app and PictureThis app, exist. Although the Florist app can only identify flowers, the PictureThis app has most of the intended features of GreenThumb, although, as GreenThumb is being developed primarily as a learning experience, this does not pose any issues.

2.1.1 System Interfaces

There are three primary components to the GreenThumb app:

  1. Frontend: the component of the GreenThumb application responsible for the GUI and integration with the user’s Android device.
  2. Backend: the component of the GreenThumb application responsible for running the API that allows the frontend to communicate with the database. The ML primarily occurs here.
  3. Database: the tables containing the information on each species of plant, as well as user information and user-uploaded photographs.

2.1.2 User Interface

  1. Splash: Page that appears on launch, shows the logo while the app loads
  2. Login: Appears on first launch, or if no Google account is registered. This page prompts the user to log in through google.
  3. Main Menu: Has buttons to link to the scan and search pages, as well as a button on top to access settings. Displays a white rose behind the logo in the background.
  4. Settings: Has an about section, a link to the login page if a user would like to change accounts, and a font size slider.
  5. Scan page: Allows the user to scan plants and submit them to be identified.
  6. Search page: Allows users to search for different types of plants.
  7. Plant Bio page: Has info about the plant a user has scanned or searched for. Is linked to from the search or scan pages upon a successful scan or a search.

Note: none of the above images represent the final GUI, but are to be used a example of what one could look like, and will change over the course of the project based on input from users/team members.)

2.1.3 Communications Interfaces

GreenThumb uses Google’s Sign In integration to include user accounts.

2.2 Product Functions

The intended primary functions of GreenThumb are as follows:

  • Identify plants and access corresponding species information by uploading an image.
  • Posting images of a species to its database page.
  • Upvoting/downvoting images on a plant’s page, to filter out low-quality images without admin intervention, allowing useful pictures to appear first.
  • Search the plant database for species that meet certain parameters.
  • Saving a user’s viewing history for quick access later.

2.3 User Characteristics

The intended audience for GreenThumb is individuals with an interest in identifying plants, including campers, who may wish to learn the possible dangers of a plant, parents, who could use the app as an educational tool for their children, and gardeners. There will be three primary types of users:

  1. Guest: A user with limited privileges within GreenThumb. They may upload photos to have their plants identified, but these photos will not be posted to the corresponding species’ page. They do not have access to photo upvoting, downvoting, or reporting options. A guest can become a registered user by logging into GreenThumb with their Google account.
  1. Registered User: A user with the privileges of a guest, along with the ability to post photos to the page for a species of plant, and upvote, downvote, or report the photos of other users. A registered user can remove their own photos.
  1. Admin: A user with the privileges of a registered user, along with the ability to remove the photos of other users, add a species to the database, update an existing species in the database, and remove a species from the database.

2.4 Constraints

Presently, no iOS version of GreenThumb is planned, as too few team members have access to a personal Mac device on which to write code. Should this change, an iOS version will likely be developed. No use of a mapping services such as Google Maps is planned, either.

2.5 Assumptions, Dependencies

It is assumed that all devices using GreenThumb have a camera with which to take photographs, and have access to at least the Android 5.0 – Lollipop operating system. Furthermore, a user must have access to the internet at all times while using GreenThumb.

2.6 Apportioning of Requirements

The following features may appear in a later version of GreenThumb:

  • Creation of an iOS version of the application.
  • Gardening information for applicable species.
  • Supplier information for seeds.
  • Location pinning for plant types.

3. Specific Requirements

3.1 Functions

Guests can use the following functions:

  • Log into their existing Google account.
  • Upload a plant photo and receive the identification.
  • Browse the plant database.

Registered users can use all the functions of a guest and:

  • Post photos to the corresponding species page.
  • Delete own photos.
  • Upvote/downvote/report photos.
  • View browsing history.
  • Clear browsing history.

Admins can use all the functions of a registered user and:

  • Delete another user’s photos.
  • Add a new species page.
  • Update an existing species page.
  • Delete an existing species page.

3.2 Performance Requirements

The app should be able to scale and handle simultaneous interactions with the database. Most of the images should be able to be predicted in seconds, though the higher the resolution the image is and the slower the devices hardware is the longer it will take. The responsiveness of the app is dependant on the speed of the user's internet connection, as the requested data must be fetched from the server.
